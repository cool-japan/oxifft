# Developer Guide: Adding New SIMD Backends

This guide explains how to add a new SIMD backend to OxiFFT's abstraction layer.

## SIMD Architecture Overview

OxiFFT's SIMD layer has three layers:

```
detect_simd_level()          — runtime CPU feature detection (detect.rs)
SimdVector / SimdComplex     — trait API (traits.rs)
Sse2F64, Avx2F64, NeonF64…  — concrete backend types (sse2.rs, avx2.rs, neon.rs…)
```

### Traits (`oxifft/src/simd/traits.rs`)

`SimdVector` is the base trait:

```rust
pub trait SimdVector: Copy + Clone + Send + Sync + Sized {
    type Scalar: Float;
    const LANES: usize;
    fn splat(value: Self::Scalar) -> Self;
    unsafe fn load_aligned(ptr: *const Self::Scalar) -> Self;
    unsafe fn load_unaligned(ptr: *const Self::Scalar) -> Self;
    unsafe fn store_aligned(self, ptr: *mut Self::Scalar);
    unsafe fn store_unaligned(self, ptr: *mut Self::Scalar);
    fn add(self, other: Self) -> Self;
    fn sub(self, other: Self) -> Self;
    fn mul(self, other: Self) -> Self;
    fn div(self, other: Self) -> Self;
    fn fmadd(self, a: Self, b: Self) -> Self { … }  // default: mul+add
}
```

`SimdComplex` extends it with complex arithmetic:

```rust
pub trait SimdComplex: SimdVector {
    fn cmul(self, other: Self) -> Self;        // (a+bi)(c+di)
    fn cmul_conj(self, other: Self) -> Self;   // (a+bi)(c-di)
    fn butterfly(a: Self, b: Self) -> (Self, Self) { (a.add(b), a.sub(b)) }
}
```

### Backend Files

| File | ISA | LANES (f64/f32) |
|------|-----|-----------------|
| `scalar.rs` | none | 1 / 1 |
| `sse2.rs` | SSE2 (x86_64) | 2 / 4 |
| `avx.rs` | AVX (x86_64) | 4 / 8 |
| `avx2.rs` | AVX2+FMA (x86_64) | 4 / 8 |
| `avx512.rs` | AVX-512F (x86_64) | 8 / 16 |
| `neon.rs` | NEON (aarch64) | 2 / 4 |
| `sve.rs` | SVE (aarch64) | scalable |

### Runtime Detection (`oxifft/src/simd/detect.rs`)

`detect_simd_level()` returns a `SimdLevel` enum. On x86_64 it probes via
`is_x86_feature_detected!`. On aarch64 NEON is unconditional.

Probe order on x86_64: `Avx512` > `Avx2` > `Avx` > `Sse2` > `Scalar`.

## Adding a New SIMD Backend

### Step 1: Create `oxifft/src/simd/my_backend.rs`

Follow the pattern in `sse2.rs`:

```rust
use super::traits::{SimdComplex, SimdVector};
use core::arch::TARGET_ARCH::*;

#[derive(Copy, Clone, Debug)]
#[repr(transparent)]
pub struct MyBackendF64(pub __my_vec_f64);

unsafe impl Send for MyBackendF64 {}
unsafe impl Sync for MyBackendF64 {}

impl SimdVector for MyBackendF64 {
    type Scalar = f64;
    const LANES: usize = N;

    fn splat(value: f64) -> Self { unsafe { Self(intrinsic_splat(value)) } }

    /// # Safety
    /// See [`SimdVector::load_aligned`] for the full safety contract.
    unsafe fn load_aligned(ptr: *const f64) -> Self {
        unsafe { Self(intrinsic_load_aligned(ptr)) }
    }
    // … load_unaligned, store_aligned, store_unaligned, add, sub, mul, div
}

impl SimdComplex for MyBackendF64 {
    fn cmul(self, other: Self) -> Self {
        // result.re = self.re * other.re - self.im * other.im
        // result.im = self.re * other.im + self.im * other.re
        // Use the shuffle+sign-flip pattern from sse2.rs, or FMA from avx2.rs
        unsafe { … }
    }
    fn cmul_conj(self, other: Self) -> Self { … }
}
```

### Step 2: Register in `simd/mod.rs`

```rust
#[cfg(target_arch = "my_arch")]
mod my_backend;

#[cfg(target_arch = "my_arch")]
pub use my_backend::{MyBackendF64, MyBackendF32};
```

Add a `SimdLevel` variant in `detect.rs` and a detection branch in
`detect_simd_level()`.

### Step 3: Wire into codelet dispatch

The dispatch for sizes 2, 4, 8 is generated by `oxifft-codegen-impl/src/gen_simd/`.
For the hand-tuned paths (sizes 16–4096 on x86_64), dispatch lives in
`oxifft/src/dft/codelets/simd/mod.rs`. Each `notw_N_dispatch` function has an
if-chain ordered best-to-worst. Add your check in the correct position:

```rust
pub fn notw_8_dispatch<T: Float>(x: &mut [Complex<T>], sign: i32) {
    if TypeId::of::<T>() == TypeId::of::<f64>() {
        let x_f64 = unsafe { /* cast */ };
        #[cfg(target_arch = "my_arch")]
        {
            if my_runtime_check() { my_backend_notw_8_f64(x_f64, sign); return; }
        }
        // … existing fallbacks
    }
}
```

### Step 4: Add tests in `my_backend.rs`

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_cmul_f64() {
        // (1 + 2i) * (3 + 4i) = -5 + 10i
        let a = unsafe { MyBackendF64::load_unaligned([1.0, 2.0, …].as_ptr()) };
        let b = unsafe { MyBackendF64::load_unaligned([3.0, 4.0, …].as_ptr()) };
        let c = a.cmul(b);
        let mut out = [0.0_f64; 2 * N];
        unsafe { c.store_unaligned(out.as_mut_ptr()) };
        assert!((out[0] - (-5.0)).abs() < 1e-12);
        assert!((out[1] - 10.0).abs() < 1e-12);
    }
}
```

## Codelet Generation

Small sizes (2, 4, 8, 16) are generated by `oxifft-codegen-impl/src/gen_simd/`:

| Generator | Emits |
|-----------|-------|
| `scalar.rs` | generic scalar fallback |
| `sse2.rs` | `codelet_simd_N_sse2_f64/f32` |
| `avx.rs` | `codelet_simd_N_avx_f64` |
| `avx2.rs` | `codelet_simd_N_avx2_f64/f32` (FMA) |
| `avx512.rs` | `codelet_simd_N_avx512_f64/f32` |
| `neon.rs` | `codelet_simd_N_neon_f64/f32` |

Each `gen_simd_size_N()` function in `mod.rs` calls all per-backend generators and
concatenates their `TokenStream`s. `gen_dispatcher()` produces the public dispatch
function that probes features at runtime and calls the matching inner function.

To add a new backend to the codegen:
1. Create `gen_simd/my_backend.rs` and implement `gen_my_backend_size_N()`.
2. Call it inside each `gen_simd_size_N()` and include the token stream in the `quote!`.
3. Add a detection branch in `gen_dispatcher()` in priority order.

## Safety Requirements

### `#[target_feature(enable = "...")]`

Every `unsafe fn` that calls ISA intrinsics **must** carry this attribute. Without it:
- The compiler may not emit the specialized instructions.
- Calling the function on a CPU that lacks the feature is undefined behavior.

```rust
#[target_feature(enable = "avx2,fma")]
unsafe fn butterfly_avx2(data: &mut [f64], sign: i32) { … }
```

### Alignment

| ISA | `load_aligned` / `store_aligned` requirement |
|-----|---------------------------------------------|
| SSE2 / NEON | 16-byte aligned |
| AVX / AVX2 | 32-byte aligned |
| AVX-512 | 64-byte aligned |

`load_unaligned` / `store_unaligned` have no alignment requirement. OxiFFT uses
unaligned loads internally because the allocator alignment is not guaranteed.

### `# Safety` doc comments

Every `unsafe fn` must document its safety invariants. The existing codebase uses
`See [`SimdVector::load_aligned`] for the full safety contract.` to delegate to the
trait-level documentation.

## Performance Testing

```
cargo bench -p oxifft-bench
```

The suite in `oxifft-bench/benches/` covers sizes 2 to 65536. Add a dedicated group
to compare your backend against the tier below it. Expected hierarchy for a 1024-point
f64 FFT:

```
Scalar < SSE2/NEON < AVX < AVX2+FMA < AVX-512
```

If your backend doesn't outperform the previous tier, verify:
1. `#[target_feature]` annotations are correct on every inner function.
2. The SIMD operations are not being scalarized (inspect with `cargo asm`).
3. No spurious bounds checks are blocking vectorization.
